## 2024-05-24 - Secure Error Handling API Responses
**Vulnerability:** Information Exposure (CWE-200). Express API endpoints returned `res.status(500).json({ error: error.message })`.
**Learning:** Returning `error.message` on 500 status codes directly to clients exposes sensitive internal details like stack traces or database errors, which can aid malicious actors in understanding system internals.
**Prevention:** Always implement a generic catch-all error response (e.g., `res.status(500).json({ error: 'Internal server error' })`) and rely exclusively on server-side logging (`console.error`) to capture the actual error details for debugging.
## 2026-03-30 - Missing Authentication & Authorization in Backend Proxy Endpoints\n**Vulnerability:** The `/api/admin/users/:uid/role` endpoint accepted an unverified `idToken` from the client and proxied it directly to the Firestore REST API without checking if the token actually belonged to an administrator. This allowed any user with a valid Firebase Auth token to spoof the `idToken` header and update their own or other users' roles (Privilege Escalation) because the backend was effectively acting as a confused deputy, bypassing standard app-level admin checks.\n**Learning:** Relying purely on client-side routing (e.g., hiding UI elements) or assuming backend REST calls are automatically protected by Firestore rules is unsafe, especially if the backend simply forwards a raw token. Server-side validation is mandatory.\n**Prevention:** Admin backend endpoints must natively verify `idToken` via `getAuth().verifyIdToken()` AND explicitly query the database to check the user's role or email.

## 2026-04-03 - Hardcoded Administrative Email Backdoor
**Vulnerability:** Privilege Escalation / Backdoor (CWE-798). A specific email address ('athenarosiejohnson@gmail.com') was hardcoded across multiple frontend views and backend API endpoints to bypass Role-Based Access Control (RBAC) and grant administrative privileges.
**Learning:** Hardcoded emails intended for testing, debugging, or specific users act as silent backdoors in production. They bypass the standard database-driven RBAC mechanisms, creating an unrevokable and unauditable escalation path.
**Prevention:** Rely strictly on database-driven Role-Based Access Control (RBAC) and the user's role property for authorization decisions. Never hardcode specific emails or user IDs to bypass these checks.
